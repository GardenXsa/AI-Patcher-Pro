Инструмент для безопасного применения кодовых патчей, сгенерированных языковыми моделями (ChatGPT, Claude, DeepSeek и др.). Работает как промежуточный слой между выводом LLM и файловой системой: принимает JSON-патч, находит целевой код с помощью 5-уровневого алгоритма нечёткого поиска, показывает diff-превью и применяет изменения с автоматическим бэкапом. Начиная с версии 3.0, патчи могут содержать консольные команды, которые выполняются в безопасном окружении с обнаружением опасных паттернов и подтверждением пользователем.

## Проблема

Языковые модели генерируют кодовые патчи в формате JSON, но при этом регулярно допускают ошибки:

- Неточное воспроизведение существующего кода в поле `search` (лишние пробелы, другие кавычки, пропущенные строки)
- Висячие запятые в JSON (trailing commas)
- Альтернативные имена полей (`path`/`file`, `action`/`op`, `search`/`original`)
- Вставка JSON внутри Markdown-блоков с пояснительным текстом

Помимо модификации файлов, патчи часто требуют выполнения консольных команд -- установка зависимостей, запуск миграций, перезапуск сервисов. До версии 3.0 эти команды пользователю приходилось выполнять вручную, что замедляло рабочий процесс и приводило к ошибкам, когда шаги пропускались или выполнялись в неправильном порядке.

Ручное применение таких патчей приводит к ошибкам. AI Patcher Pro решает эту проблему автоматическим нечётким поиском, валидацией каждого изменения и встроенным выполнением консольных команд с контролем безопасности.

### Демонстрация нечёткого поиска

LLM часто искажает исходный код. Поисковый движок компенсирует это:

```
 ФАЙЛ НА ДИСКЕ:                              ИИ ПОЛАГАЕТ (search):
                                             (пробелы искажены, кавычки другие)

 1  def calculate(x, y):                    1  def calculate( x , y ):
 2      result = x + y                      2      result = x+y
 3      if result > 0:                      3      if result>0:
 4          print("positive")               4          print('positive')
 5      return result                       5      return result

 УРОВЕНЬ 1 (Точное):     НЕ НАЙДЕНО  -- пробелы и кавычки отличаются
 УРОВЕНЬ 2 (Якоря):      НЕ НАЙДЕНО  -- строк слишком мало
 УРОВЕНЬ 3 (Нормализ.):  НАЙДЕНО     -- после удаления пробелов и унификации
                                      кавычек строки идентичны

 normalize("  result = x + y  ") --> "result=x+y"
 normalize("  result = x+y  ")   --> "result=x+y"

 Нормализованные строки совпадают:
 "defcalculate(x,y):result=x+yifresult>0:print("positive")returnresult"
 "defcalculate(x,y):result=x+yifresult>0:print("positive")returnresult"

 Результат: точный фрагмент из файла возвращён для замены
```

## Принцип работы

```
+----------+     +---------------+     +----------------+     +----------+
|  Ответ   |     |  Парсер JSON  |     |  Поисковый     |     |  Diff    |
|  LLM     | --> |  (extract +   | --> |  движок        | --> |  превью  |
|          |     |   normalize)  |     |  (5 уровней)   |     |          |
+----------+     +---------------+     +----------------+     +----------+
                                                                   |
                                                                   v
                                                          +----------------+
                                                          |  Виртуальная   |
                                                          |  файловая      |
                                                          |  система       |
                                                          +----------------+
                                                                   |
                                          +------------------------+------------------------+------------------------+
                                          |                        |                        |                        |
                                          v                        v                        v                        v
                                   +-----------+           +-----------+           +-----------+           +-----------+
                                   |  Бэкап    |           |  Запись   |           |  Команды  |           |  Отчёт    |
                                   |  файлов   |           |  на диск  |           |  (cmds)   |           |  об ошиб. |
                                   +-----------+           +-----------+           +-----------+           +-----------+
                                                                                          |
                                                                        +----------------+----------------+
                                                                        |                                 |
                                                                        v                                 v
                                                               after_analysis                      after_apply
                                                            (до записи файлов)              (после записи файлов)
```

### Жизненный цикл патча

1. **Ввод** -- пользователь вставляет ответ LLM или перетаскивает файл в окно приложения
2. **Парсинг** -- извлечение JSON из Markdown, автофикс висячих запятых, нормализация полей и команд
3. **Поиск** -- для каждой операции выполняется 5-уровневый поиск целевого кода в файле
4. **Оценка** -- операция выполняется в виртуальной файловой системе (в памяти), вычисляется diff
5. **Команды after_analysis** -- выполняются команды с флагом `after_analysis` (до применения изменений на диск, но после анализа). Типичные примеры: установка зависимостей, проверка окружения
6. **Просмотр** -- пользователь видит карточку каждой операции: статус, метод поиска, diff-превью, а также список команд с их статусом
7. **Применение** -- после подтверждения: бэкап оригинальных файлов, запись изменений на диск
8. **Команды after_apply** -- выполняются команды с флагом `after_apply` (после записи изменений). Типичные примеры: запуск тестов, сборка проекта, перезапуск сервиса

### Виртуальная файловая система

Все операции вычисляются в памяти (`memory_files: Dict[str, str]`), без записи на диск. Это позволяет:

- Накапливать несколько операций над одним файлом корректно (каждая следующая видит результат предыдущей)
- Отменять и редактировать операции без побочных эффектов
- Гарантировать атомарность: либо применяются все операции, либо ни одной

### Разделение ответственности

| Слой | Модули | Ответственность |
|------|--------|----------------|
| **Core** | search, patcher, processor, command_utils, command_executor | Бизнес-логика, не зависит от GUI |
| **GUI** | main_window, operation_card, ... | Отображение, взаимодействие с пользователем |
| **Utils** | file_io | Вспомогательные функции ввода-вывода |

Core-модули не импортируют GUI-модули. `ProcessorThread` и `CommandExecutorThread` связывают их через сигналы Qt (`pyqtSignal`), сохраняя отзывчивость интерфейса. Модуль `command_utils.py` не зависит от Qt и может использоваться в парсере и тестах без GUI, тогда как `command_executor.py` наследует `QThread` для асинхронного выполнения.

## Алгоритмы поиска

Поисковый движок (`SearchEngine`) последовательно применяет 5 уровней поиска, от точного к нечёткому. Возвращается первый успешный результат.

### Уровень 1. Точное совпадение

Проверка вхождения подстроки `search_text` в `file_text` напрямую.

```
result = search_text in file_text
```

Если найдено -- возвращается оригинальный `search_text` как есть. Это самый быстрый и надёжный путь, не требующий никаких эвристик.

**Пример:**

```
search_text: "def hello():\n    print('hello')"
file_text:   "... def hello():\n    print('hello') ..."

Результат:  "def hello():\n    print('hello')"   (точная копия)
Метод:      "Точное совпадение"
```

### Уровень 2. Умные якоря

Если ИИ исказил середину фрагмента, но первая и последняя строки уникальны в файле, блок между ними определяется как результат.

Условия срабатывания:

```
count(first_line, file) == 1  AND  count(last_line, file) == 1
```

Вычисление границ:

```
start_idx = position(first_line, file)
end_idx   = position(last_line, file) + length(last_line) + offset_to_next_newline
```

Защита от ложных срабатываний -- блок не должен превышать утроенный размер искомого фрагмента:

```
(end_idx - start_idx) < len(search_text) * 3
```

Если перенос строки после `last_line` не найден (последняя строка файла), `end_idx` устанавливается в `len(file_text)`.

**Пример:**

```
search_text от ИИ:                          Файл на диске:

1  class Calculator:                        1  class Calculator:
2      def add WRONG STUFF HERE             2      def add(self, a, b):
3          return WRONG                      3          return a + b
4      def multiply WRONG                   4      def multiply(self, a, b):
5          return WRONG                      5          return a * b
6  # End of class                           6  # End of class

Якоря:  first_line = "class Calculator:"   (1 вхождение -- уникальна)
        last_line  = "# End of class"      (1 вхождение -- уникальна)

Результат:  строки 1-6 из файла (точный фрагмент)
Метод:      "Умный поиск по якорям (начало и конец блока)"
```

### Уровень 3. Нормализованное сравнение

Удаление всех пробельных символов и унификация кавычек:

```
normalize(text):
    for each character in text:
        if character is whitespace: skip
        if character in ('"', "'", '`'): replace with '"'
        else: keep character
    return normalized_string, index_mapping
```

После нормализации выполняется поиск подстроки:

```
file_norm = normalize(file_text)
search_norm = normalize(search_text)

if search_norm in file_norm:
    return file_text[file_map[i] : file_map[i + len(search_norm) - 1] + 1]
```

`index_mapping` -- массив, связывающий индексы нормализованной строки с оригинальной, что позволяет восстановить точные границы совпадения в исходном тексте.

**Пример:**

```
Исходный файл:                Искомый текст (от ИИ):

x = "hello"                   x='hello'
y = x + ' world'              y=x+'world'

Нормализация файла:           Нормализация поиска:
x="hello"y=x+"world"          x="hello"y=x+"world"

Нормализованные строки идентичны  -->  НАЙДЕНО
Метод: "Игнор пробелов/кавычек"
```

### Уровень 4. Построчный нечёткий поиск

Сравнение скользящим окном с порогом сходства 92%.

```
search_lines = [line.strip() for line in search_text.splitlines() if line.strip()]
window_size  = len(search_lines)

for i in range(len(file_lines) - window_size + 1):
    chunk = file_lines[i : i + window_size + 2]
    ratio = SequenceMatcher(search_lines, chunk_stripped)
    if ratio > 0.92:
        return file_lines[i : end]
```

Коэффициент сходства вычисляется алгоритмом Ratcliff/Obershelp:

```
                    2 * M
    ratio = -------------------
            len(A) + len(B)

    где M -- длина наибольшей общей подпоследовательности
```

Окно расширяется на +2 строки для устойчивости к добавленным/удалённым строкам.

**Пример:**

```
Искомый текст (5 строк):          Окно в файле (5 строк + 2 запасных):

1  def process(data):             1  def process(data):
2      result = []                2      result = []
3      for item in data:          3      for item in data:       <-- ИИ добавил
4          result.append(item)    4          if item.valid:       <-- лишнюю строку
5      return result              5              result.append(item)
                                   6      return result

ratio = 2 * 180 / (195 + 210) = 0.89  --> НИЖЕ порога 0.92

Но при расширении окна на +2 строки:
ratio = 2 * 210 / (195 + 220) = 0.98  --> ВЫШЕ порога 0.92  --> НАЙДЕНО
```

### Уровень 5. Нечёткий поиск подстрок

Скользящее окно по нормализованному тексту с порогом 80%. Возвращает до 4 кандидатов для ручного выбора.

```
step = max(1, len(search_norm) // 3)
window_extra = int(len(search_norm) * 0.3)

for i in range(0, len(file_norm) - len(search_norm) + 1, step):
    window = file_norm[i : i + len(search_norm) + window_extra]
    ratio = SequenceMatcher(search_norm, window)
    if ratio > 0.80:
        longest = find_longest_match(search_norm, window)
        candidate = file_text[file_map[i + longest.b : ...]]
        suggestions.append((ratio, candidate))

return top 4 suggestions by ratio descending
```

Размер шага `len(search_norm) // 3` обеспечивает баланс между полнотой и производительностью. Расширение окна на 30% компенсирует разницу в длине между искомым и найденным фрагментами.

**Пример:**

```
ИИ ищет (нормализованно):         Файл содержит (нормализованно):
"defprocess(data):result=[]"       "defprocess(items):output=[]"

ratio = 2 * 18 / (24 + 25) = 0.73  --> НИЖЕ 0.80

Другая позиция:
"defprocess(data):result=[]"       "defprocess(data):results=[]"

ratio = 2 * 22 / (24 + 25) = 0.90  --> ВЫШЕ 0.80  --> КАНДИДАТ

Результат: список из до 4 кандидатов с процентом сходства
для ручного выбора пользователем
```

## Формат патча

### Поддерживаемые операции

| Действие | Описание | Обязательные поля |
|----------|----------|-------------------|
| `replace` | Замена найденного фрагмента на новый | `path`, `search`, `content` |
| `insert_after` | Вставка после найденного фрагмента | `path`, `search`, `content` |
| `insert_before` | Вставка перед найденным фрагмента | `path`, `search`, `content` |
| `delete` | Удаление найденного фрагмента | `path`, `search` |
| `append` | Добавление в конец файла | `path`, `content` |
| `prepend` | Добавление в начало файла | `path`, `content` |
| `create_file` | Создание нового файла | `path`, `content` |

### Формат JSON

Массив операций:

```json
[
    {
        "path": "src/main.py",
        "action": "replace",
        "search": "def hello():\n    print('old')",
        "content": "def hello():\n    print('new')"
    },
    {
        "path": "src/utils.py",
        "action": "create_file",
        "content": "def helper():\n    pass"
    }
]
```

Или объект с метаданными и командами:

```json
{
    "patch_name": "Рефакторинг hello()",
    "operations": [
        { "path": "...", "action": "...", "search": "...", "content": "..." }
    ],
    "commands": [
        { "cmd": "pip install requests", "run": "after_apply", "description": "Установка зависимостей" }
    ]
}
```

### Альтернативные имена полей

LLM могут использовать разные имена полей. Парсер автоматически нормализует:

| Стандартное | Альтернативы |
|-------------|-------------|
| `path` | `file` |
| `action` | `op` |
| `search` | `original`, `find` |
| `content` | `text`, `code` |

### Альтернативные имена для поля commands

Поле `commands` на верхнем уровне патча также распознаётся под альтернативными именами. Парсер автоматически нормализует любое из указанных имён к стандартному `commands`:

| Стандартное | Альтернативы |
|-------------|-------------|
| `commands` | `cmds`, `exec`, `run_commands`, `post_commands` |

Это позволяет ИИ использовать привычные ему имена полей без потери функциональности. Например, ИИ может сгенерировать `"cmds": [...]` или `"post_commands": [...]`, и парсер корректно извлечёт список команд из любого варианта.

## Выполнение консольных команд

Начиная с версии 3.0, патчи могут содержать консольные команды, которые выполняются автоматически в нужный момент рабочего процесса. Это устраняет необходимость вручную копировать команды из ответа ИИ и выполнять их в терминале, что особенно важно для патчей, требующих установки зависимостей, запуска миграций или проверки результатов.

### Схема команды

Каждая команда описывается словарём со следующими полями:

```json
{
    "cmd": "pip install mypy",
    "run": "after_analysis",
    "description": "Установка проверки типов"
}
```

**Поля команды:**

| Поле | Обязательное | Описание |
|------|-------------|----------|
| `cmd` | Да | Строка команды для выполнения в оболочке |
| `run` | Нет | Время выполнения: `after_analysis` или `after_apply` (по умолчанию `after_apply`) |
| `description` | Нет | Человекочитаемое описание команды |

**Альтернативные имена полей команды:**

| Стандартное | Альтернативы |
|-------------|-------------|
| `cmd` | `command`, `exec`, `shell` |
| `run` | `timing`, `when`, `phase` |
| `description` | `desc`, `comment`, `note` |

### Краткие формы записи команд

Для удобства и совместимости с различными форматами вывода ИИ поддерживаются сокращённые формы записи:

**Строка** -- команда без уточнения времени выполнения. По умолчанию `run` устанавливается в `after_apply`:

```json
"commands": ["pip install requests", "npm run build"]
```

Эквивалентно:

```json
"commands": [
    {"cmd": "pip install requests", "run": "after_apply"},
    {"cmd": "npm run build", "run": "after_apply"}
]
```

**Массив строк** -- несколько команд в краткой форме, все выполняются после применения патча:

```json
"commands": ["pip install -r requirements.txt", "python manage.py migrate"]
```

### Время выполнения: after_analysis vs after_apply

Выбор правильного времени выполнения команды критически важен для корректной работы патча. AI Patcher Pro поддерживает два режима, каждый из которых решает свою задачу в жизненном цикле применения патча.

**`after_analysis`** -- команда выполняется после завершения парсинга и анализа патча, но до записи изменений на диск. На этом этапе файлы ещё не модифицированы, но виртуальная файловая система уже построена и diff-превью доступно. Этот режим предназначен для команд, которые должны подготовить окружение до применения изменений: установка новых зависимостей, которые понадобятся модифицированному коду; проверка наличия необходимых инструментов в системе; предзагрузка данных или ресурсов. Если команда `after_analysis` завершается с ошибкой, пользователь видит это до применения патча и может отменить операцию.

**`after_apply`** -- команда выполняется после успешной записи всех изменений на диск и создания бэкапа. На этом этапе файлы уже модифицированы, и команды могут работать с обновлённым кодом. Этот режим предназначен для команд, которые работают с результатами применения патча: запуск тестов для проверки корректности изменений; сборка проекта с новыми файлами; перезапуск сервисов, использующих модифицированный код; выполнение миграций базы данных. Если команда `after_apply` завершается с ошибкой, изменения уже записаны на диск, и пользователь может откатиться через систему бэкапов.

```
Порядок выполнения:

1. Парсинг JSON
2. Поиск и анализ операций
3. >>> Команды after_analysis <<<
4. Подтверждение пользователя
5. Бэкап файлов
6. Запись изменений на диск
7. >>> Команды after_apply <<<
```

**Когда использовать after_analysis:**

- Установка пакетов, которые импортируются в новом коде (`pip install`, `npm install`)
- Проверка версий инструментов (`python --version`, `node --version`)
- Подготовка окружения (`mkdir -p data/`, `cp config.example.json config.json`)

**Когда использовать after_apply:**

- Запуск тестов (`pytest`, `npm test`)
- Сборка проекта (`npm run build`, `python setup.py build`)
- Миграции (`python manage.py migrate`, `alembic upgrade head`)
- Перезапуск сервисов (`systemctl restart myapp`, `docker-compose restart`)

### Обнаружение опасных команд

Функция `check_dangerous_command` из модуля `command_utils.py` проверяет каждую команду на наличие опасных паттернов перед выполнением. Обнаружение работает нечувствительно к регистру и основано на списке регулярных выражений, покрывающих наиболее разрушительные команды.

**Обнаруживаемые опасные паттерны:**

| Паттерн | Описание |
|---------|----------|
| `rm -rf` | Удаление директории без подтверждения |
| `rmdir /s` | Рекурсивное удаление директории (Windows) |
| `del /s` | Рекурсивное удаление файлов (Windows) |
| `format X:` | Форматирование диска |
| `shutdown` | Выключение системы |
| `reboot` | Перезагрузка системы |
| `mkfs` | Форматирование файловой системы |
| `dd if=` | Низкоуровневое копирование диска |
| `> /dev/sd` | Прямая запись на блочное устройство |
| `chmod -R 777` | Установка прав 777 на все файлы |
| `chown -R` | Рекурсивная смена владельца |
| `sudo rm` | Удаление с правами суперпользователя |
| `taskkill /f` | Принудительное завершение процессов (Windows) |
| `reg delete` | Удаление записей реестра (Windows) |
| `net user` | Управление пользователями системы (Windows) |

При обнаружении опасного паттерна команда не блокируется автоматически, но пользователю показывается диалог подтверждения с перечислением всех обнаруженных предупреждений. Пользователь может либо подтвердить выполнение, либо отменить команду. Такой подход обеспечивает баланс между безопасностью и гибкостью: ИИ иногда генерирует команды, которые выглядят опасно, но оправданы в контексте конкретной задачи (например, `rm -rf /tmp/build_artifacts` при очистке временных файлов сборки).

### Диалог подтверждения команд

Перед выполнением любой команды (не только опасной) приложение показывает диалог с информацией о команде. Для команд с обнаруженными опасными паттернами диалог дополнительно выделяет предупреждения красным цветом и требует явного подтверждения. Это предотвращает случайное выполнение разрушительных действий, которые ИИ мог сгенерировать по ошибке или из-за галлюцинации.

### Лог выполнения команд

Все выполненные команды и их результаты отображаются в специальной панели в интерфейсе. Каждая запись содержит команду, описание, статус выполнения, stdout, stderr и код возврата. Рядом с каждой записью находится кнопка копирования, позволяющая быстро скопировать команду или её вывод для дальнейшего использования, например для передачи в отчёт об ошибках или для ручного выполнения в терминале.

### Выполнение команд

Команды выполняются через модуль `command_executor.py`, который реализован как `QThread` для неблокирующего выполнения. Каждая команда запускается через `subprocess.Popen` с параметром `shell=True`, что позволяет использовать конвейеры, перенаправления и другие возможности оболочки. Рабочей директорией для выполнения является корень проекта (workspace). Таймаут выполнения одной команды составляет 300 секунд, после чего процесс завершается принудительно. Результаты каждой команды передаются в GUI через сигнал `command_done`, а по завершении всех команд отправляется сигнал `finished_all`.

            "action": "prepend",

        },

            "path": "src/main.py",
            "action": "insert_after",
            "search": "from calculator import add",
            "content": "from calculator import subtract"
        },
        {
            "path": "tests/test_calc.py",
            "action": "create_file",
            "content": "from calculator import add\n\ndef test_add():\n    assert add(2, 3) == 5"

            "run": "after_apply",

Команда `pip install mypy` с флагом `after_analysis` выполнится до записи файлов на диск, гарантируя доступность `mypy` к моменту проверки типизации. Команды `mypy` и `pytest` с флагом `after_apply` выполнятся после успешной записи всех изменений.

Результат парсинга:
- Текст до и после JSON отбрасывается
- Извлечён массив из 2 операций
- patch_name: "Сборный патч" (автоматическое имя при отсутствии метаданных)
```

### Поддержка нескольких JSON-блоков

Если ИИ генерирует несколько блоков, они склеиваются:

```
Входной текст:
+-------------------------------------------------------------------+
| Сначала исправим баг:                                             |
| ```json                                                           |
| [{ "action": "replace", "path": "bug.py", ... }]                  |
| ```                                                               |
| А теперь добавим фичу:                                            |
| ```json                                                           |
| [{ "action": "create_file", "path": "feature.py", ... }]          |
| ```                                                               |
+-------------------------------------------------------------------+

Результат: 2 операции из разных блоков объединены в один патч
```

### Автофикс висячих запятых

LLM часто генерируют невалидный JSON с висячими запятыми. Парсер автоматически удаляет их:

```
// До фикса (невалидный JSON)
{
    "action": "replace",
    "path": "main.py",
    "search": "old code",
    "content": "new code",    <-- висячая запятая
}

// После фикса
{
    "action": "replace",
    "path": "main.py",
    "search": "old code",
    "content": "new code"
}
```

Регулярное выражение: `re.sub(r',(\s*[\]}])', r'\1', block)`

+---------------------------+   +---------------------------+
| {                         |   | {                         |

|   "op": "replace",        |   |   "action": "replace",    |
|   "original": "old_code", |   |   "search": "old_code",   |

## Безопасность

### Защита от Path Traversal

Функция `secure_path_join` проверяет, что результирующий путь находится внутри рабочей директории:

```
base_dir  = os.path.abspath(workspace)
target    = os.path.normpath(os.path.abspath(os.path.join(base_dir, relative_path)))

if target != base_dir AND NOT target.startswith(base_dir + os.sep):
    raise PermissionError
```

Использование `os.sep` после `base_dir` предотвращает совпадение префиксов (например, `/home/user` не совпадает с `/home/user10`).

Блокируемые атаки:

```
Атака                          Резолвящийся путь           Результат
---------------------------------------------------------------------------
../../etc/passwd               /etc/passwd                 PermissionError
/etc/passwd                    /etc/passwd                 PermissionError
./../../sensitive              /home/outside/sensitive     PermissionError
../other_project/main.py       /home/other_project/...     PermissionError

Нормальный запрос:
src/main.py                    /workspace/src/main.py      OK
```

### Замена только первого вхождения

Критически важное отличие от `str.replace()`, который заменяет все вхождения:

```python
# str.replace() -- ОПАСНО для патчинга
text = "import os\nimport sys\nimport os"
text.replace("import os", "import pathlib")
# Результат: "import pathlib\nimport sys\nimport pathlib"  -- ДВА вхождения!

# replace_first_occurrence() -- БЕЗОПАСНО
replace_first_occurrence(text, "import os", "import pathlib")
# Результат: "import pathlib\nimport sys\nimport os"  -- ТОЛЬКО первое
```

При наличии повторяющихся фрагментов кода (одинаковые импорты, паттерны) замена всех вхождений приводила к непреднамеренному повреждению файлов.

### Мульти-кодировочное чтение

Файлы читаются с фоллбэком по кодировкам: `utf-8` -> `cp1251` -> `latin-1`. Это необходимо для работы с legacy-файлами на Windows, где всё ещё используется cp1251.

```
Попытка 1: utf-8    --> UnicodeDecodeError
Попытка 2: cp1251   --> Успешно прочитано
```

### Обнаружение повторного применения

Если `search` не найден, но `content` уже присутствует в файле, операция помечается как `already_applied` вместо ошибки:

```
Сценарий:
1. ИИ сгенерировал патч
2. Пользователь применил патч
3. ИИ повторно сгенерировал тот же патч (галлюцинация)

Поиск:   search не найден (старый код уже заменён)
Проверка: content уже в файле
Результат: ПРОПУСК -- "Изменения уже присутствуют"
```

### Обнаружение опасных команд

Функция `check_dangerous_command` проверяет каждую команду из патча на наличие деструктивных паттернов (см. раздел [Выполнение консольных команд](#выполнение-консольных-команд)). Обнаружение работает до выполнения команды и не блокирует её автоматически, но требует явного подтверждения пользователя через диалог. Это двухуровневая защита: сначала автоматическое обнаружение паттернов, затем ручное подтверждение. Подход предотвращает как случайное выполнение опасных команд по ошибке ИИ, так и чрезмерное ограничение функциональности, когда команда формально опасна, но оправдана в контексте задачи.

## Режим администратора

Начиная с версии 3.0, AI Patcher Pro требует запуска с правами администратора (Windows) или root (Linux/macOS). Это обязательное требование связано с новой функциональностью выполнения консольных команд: многие команды, которые ИИ генерирует для проекта, требуют повышенных привилегий (установка системных пакетов, изменение конфигурации, перезапуск сервисов). Без прав администратора такие команды завершались бы с ошибкой `PermissionError`, и пользователю приходилось бы перезапускать приложение вручную.

### Почему требуются права администратора

Выполнение консольных команд через `subprocess.Popen` с `shell=True` наследует привилегии процесса. Если приложение запущено без прав администратора, команды вроде `pip install`, `systemctl restart` или запись в защищённые директории завершатся с ошибкой. Требование повышения привилегий на старте гарантирует, что все команды из патча будут выполнены корректно без необходимости прерывать рабочий процесс.

### Поведение на Windows

При запуске на Windows приложение проверяет привилегии через Win32 API `IsUserAnAdmin()` из `shell32.dll`. Если права администратора отсутствуют, отображается диалоговое окно с предложением перезапустить приложение с повышенными привилегиями. При согласии пользователя вызывается `ShellExecuteW` с глаголом `runas`, который показывает стандартный системный диалог UAC (User Account Control) и перезапускает приложение от имени администратора. При отказе приложение завершает работу с кодом 1.

```
Проверка: ctypes.windll.shell32.IsUserAnAdmin() != 0

Если False:
  --> Диалог: "AI Patcher Pro требует запуска от имени администратора
               для корректного выполнения консольных команд.
               Перезапустить с правами администратора?"
  --> Да:  ShellExecuteW(None, "runas", sys.executable, params, None, 1)
  --> Нет: sys.exit(1)
```

### Поведение на Linux/macOS

На Unix-системах приложение проверяет эффективный идентификатор пользователя через `os.geteuid()`. Если значение отлично от 0 (не root), отображается диалоговое окно с инструкцией перезапустить приложение через `sudo`. Автоматический перезапуск с повышением привилегий на Unix не реализован из соображений безопасности: `sudo` требует ввода пароля в терминале, и автоматизация этого процесса через GUI считается небезопасной практикой.

```
Проверка: os.geteuid() == 0

Если False:
  --> Диалог: "AI Patcher Pro требует запуска с правами суперпользователя
               для корректного выполнения консольных команд.
               Запустите:
                 sudo python3 -m ai_patcher_pro"
  --> OK: sys.exit(1)
```

## Системный промпт для ИИ

Этот раздел содержит готовый системный промпт, который можно скопировать и вставить в настройки любой языковой модели (ChatGPT, Claude, DeepSeek и др.) для генерации патчей, совместимых с AI Patcher Pro. Промпт описывает полный формат JSON-патча, все поддерживаемые операции, схему команд и альтернативные имена полей. Скопируйте содержимое блока ниже и вставьте его в поле системного промпта вашего ИИ-ассистента.

```text
Формат ответа для AI Patcher Pro -- JSON-патч.

Структура патча (объект верхнего уровня):
{
  "patch_name": "string -- необязательное имя патча",
  "operations": [ ... ],
  "commands": [ ... ]
}

Допустимые имена поля commands: commands, cmds, exec, run_commands, post_commands.

Каждая операция в массиве operations:
{
  "path": "относительный путь к файлу от корня проекта",
  "action": "replace | insert_after | insert_before | delete | append | prepend | create_file",
  "search": "фрагмент кода для поиска (обязателен для replace, insert_after, insert_before, delete)",
  "content": "новый код (обязателен для replace, insert_after, insert_before, append, prepend, create_file)"
}

Действия:
- replace: заменить найденный фрагмент (search) на content
- insert_after: вставить content после найденного фрагмента
- insert_before: вставить content перед найденным фрагментом
- delete: удалить найденный фрагмент (content не нужен)
- append: добавить content в конец файла (search не нужен)
- prepend: добавить content в начало файла (search не нужен)
- create_file: создать новый файл с содержимым content (search не нужен)

Альтернативные имена полей операции:
- path -> file
- action -> op
- search -> original, find
- content -> text, code

Каждая команда в массиве commands:
{
  "cmd": "строка команды для выполнения в оболочке",
  "run": "after_analysis | after_apply",
  "description": "необязательное описание команды"
}

Альтернативные имена полей команды:
- cmd -> command, exec, shell
- run -> timing, when, phase
- description -> desc, comment, note

Краткие формы команд:
- Строка: "pip install requests" эквивалентно {"cmd": "pip install requests", "run": "after_apply"}
- Массив строк: ["npm install", "npm run build"]

Время выполнения (run):
- after_analysis: выполнить после парсинга и анализа, но до записи файлов на диск.
  Использовать для подготовки окружения: установка зависимостей, проверка версий.
- after_apply: выполнить после записи изменений на диск.
  Использовать для проверки результатов: запуск тестов, сборка, миграции.
- По умолчанию: after_apply.

Порядок выполнения:
1. Парсинг JSON
2. Поиск и анализ операций
3. Команды after_analysis
4. Подтверждение пользователя
5. Бэкап и запись файлов
6. Команды after_apply

Правила:
- Используйте \n для перевода строк внутри строковых значений.
- Не используйте висячие запятые (trailing commas).
- Поле path всегда содержит относительный путь от корня проекта.
- Для replace/insert_after/insert_before/delete в search указывайте точный фрагмент существующего кода.
- Поиск поддерживает нечёткое совпадение: незначительные отличия в пробелах и кавычках допустимы.
- Если патч уже был применён, повторное применение будет пропущено автоматически.
```
